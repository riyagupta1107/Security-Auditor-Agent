# Security Audit Report for `test-codebase/db.js`

**Date:** 2026-04-20

## Summary
The `db.js` file contains two critical security issues:
1. **Hard‑coded credentials** – a plaintext password is stored directly in source code.
2. **SQL Injection vulnerability** – a SQL query is built by concatenating user‑controlled input (`userId`) without any sanitisation or parameterisation.

Both issues could lead to unauthorized data access, data leakage, and full system compromise.

---

## Detailed Findings

### 1. Hard‑coded Password
```js
const password = 'SuperSecretPassword123!';
```
- **Risk:** Storing secrets in source code makes them discoverable by anyone with repository access, including attackers who may gain read‑only access. It also increases the chance of accidental leakage (e.g., committing to a public repo).
- **Best Practice:** Use environment variables, a secrets manager (e.g., Vault, AWS Secrets Manager), or a configuration file that is excluded from version control (`.gitignore`).

### 2. SQL Injection Vulnerability
```js
const query = 'SELECT * FROM users WHERE id = ' + userId;
```
- **Risk:** The `userId` value is concatenated directly into the SQL statement. If an attacker can control `userId`, they can inject arbitrary SQL (e.g., `1; DROP TABLE users;`). This can lead to data exfiltration, modification, or deletion.
- **Best Practice:** Use parameterised queries / prepared statements provided by the database driver (e.g., `db.query('SELECT * FROM users WHERE id = ?', [userId])`). Validate and sanitise all inputs, and enforce type checking (ensure `userId` is a number).

---

## Recommendations
1. **Remove hard‑coded secrets**
   - Store the password in an environment variable, e.g., `process.env.DB_PASSWORD`.
   - Add a `.env.example` file to document required variables without exposing values.
2. **Mitigate SQL injection**
   - Switch to parameterised queries or an ORM that automatically escapes inputs.
   - Validate `userId` (e.g., `Number.isInteger(userId)` or use a schema validator).
3. **General hardening**
   - Review the entire codebase for other instances of hard‑coded credentials or string‑concatenated queries.
   - Enable linting rules that flag insecure patterns (e.g., `no-eval`, `no-inline-credentials`).
   - Consider adding a secret‑scanning CI step (e.g., GitGuardian, truffleHog).

---

## Conclusion
The current implementation of `db.js` poses a high security risk due to exposed credentials and an exploitable SQL injection vector. Addressing the recommendations above will significantly improve the security posture of the application.
